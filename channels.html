Prelude: The Collatz Conjecture

Starting with an integer n, repeat the following:

  • If n is even, divide it by 2.
  • If n is odd, multiply it by 3 and add 1.

Examples: 4 -> 2 -> 1 6 -> 3 -> 10 -> 5 -> 16 -> 8 -> 4 -> 2 -> 1 7 -> 22 -> 11 -> 34 -> 17 -> 52 -> 26 -> 13 -> 40 -> 20 -> 10 -> 5 -> 16 -> 8 -> 4 -> 2 -> 1

Conjecture: All values end up at 1.

2

The collatz function

// collatz returns the number of steps to get to 1 from n using the Collatz
// sequence.
1func collatz(n int) int {
2   count := 0
3   for n > 1 {
4      if n%2 == 0 {
5         n /= 2
6      } else {
7         n = 3*n + 1
8      }
9      count++
10   }
11   return count
12}
3

Passing a value between goroutines

We can pass a value between goroutines with a WaitGroup.

1var wg sync.WaitGroup
2var v int
3wg.Go(func() { v = collatz(7) })
4wg.Wait()
5fmt.Println(v)

But there is a more flexible way: channels.

4

Unbuffered channels

An unbuffered channel lets two goroutines rendezvous.

They wait for each other.

1c := make(chan int) // create a channel

2go func() { c <- collatz(7) }() // send to c

3v := <-c // receive from c

4fmt.Println(v)

It doesn't matter which happens first, the send or the receive.

Sends and receives are thread-safe.

You can have many senders, and many receivers.

1c := make(chan int)
2for i := range 3 {
3   go func() { c <- collatz(i) }()
4}
5for range 3 {
6   go func() {
7      fmt.Println(<-c)

The select statement

Task: run a goroutine, timing out after a fixed duration.

Use time.After for timeouts.

1c := make(chan int)
2go func() { c <- collatz(7) }()
3select {
4case v := <-c:
5   fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7   fmt.Println("timed out")
8}

The timeout logic is right, but something else is wrong...

Goroutine leaks

1c := make(chan int)
2go func() { c <- collatz(7) }()
3select {
4case v := <-c:
5   fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7   fmt.Println("timed out")
8}
// program continues
What happens to the first goroutine if there is a timeout?
  1. time.After case executes
  2. select finishes
  3. goroutine tries to send to c
  • The GC does not collect c: there is still a reference to it.
  • The GC does not collect goroutines: they must terminate.

Buffered channels

1c := make(chan int, 1) // cap(c) == 1
2go func() { c <- collatz(7) }()
3select {
4case v := <-c:
5   fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7   fmt.Println("timed out")
8}
// program continues
  • The size of the queue is the capacity of the channel.
  • Sending enqueues, blocks if full.
  • Receiving dequeues, blocks if empty.
  • Sender and receiver don't have to rendezvous.
Now what happens to the first goroutine if there is a timeout?
  1. time.After case executes
  2. select finishes
  3. goroutine tries to send to c
  4. value is enqueued in channel
  5. goroutine exits

No leaks: goroutine terminates

No garbage: Nothing refers to c

Non-blocking select

Let's implement an in-process notification service:

// sendNotification optionally sends a notification without
// blocking the current goroutine.
// To avoid blocking, the notification might be dropped.
1func sendNotification(string) {}

// receiveNotification receives a notification if there is one.
// It returns "" if there isn't.
// It never blocks.
2func receiveNotification() string { return "" }
11

Notifications: first attempt

1var nc = make(chan string)

2func sendNotification(s string) { nc <- s }

3func receiveNotification() string { return <-nc }
Does this implementation satisfy our spec?

No. Both functions can block.

What if we add buffering?

It helps until the buffer fills (send) or is empty (receive). Then they block again.

12

Notifications: solution

The default case of a select executes when no channel operations are ready.

1var nc = make(chan string, 10)

2func sendNotification(s string) {
3   select {
4   case nc <- s:
5   default: // if we can't send, drop s
6   }
7}

8func receiveNotification() string {
9   select {
10   case s := <-nc:
11      return s
12   default:
13      return ""
14   }
15}

Closed channels

Close a channel when it will never be sent to again.

1type node struct {
2   val         int
3   left, right *node
4}

5func (n *node) values() chan int {
6   c := make(chan int)
7   go func() {
8      sendValues(n, c)
9      close(c)
10   }()
11   return c
12}

 

1func sendValues(n *node, ch chan int) {
2   if n == nil {
3      return
4   }
5   sendValues(n.left, ch)
6   ch <- n.val
7   sendValues(n.right, ch)
8}

In modern Go, we would use an iterator.

close broadcasts

close affects every receiver

1c := make(chan int)
2var wg sync.WaitGroup
3wg.Go(func() { c <- 0 })
4wg.Go(func() { fmt.Println(<-c) })
5wg.Go(func() { fmt.Println(<-c) })
6wg.Wait()
What does this do?

Prints 0, then hangs.

1c := make(chan int)
2var wg sync.WaitGroup
3wg.Go(func() { close(c) })
4wg.Go(func() { fmt.Println(<-c) })
5wg.Go(func() { fmt.Println(<-c) })
6wg.Wait()
What does this do?

Prints 0 twice, then finishes.

Interrupting goroutines

1c := make(chan int, 1)
2go func() { c <- collatz(n) }()
3select {
4case v := <-c:
5   fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7   fmt.Println("timed out")
8}
// program continues

 

collatz(n) will keep running until it's done, consuming resources.

You can't interrupt an arbitrary goroutine.

It has to check.

Asking a goroutine to stop

1c := make(chan int, 1)
2done := make(chan struct{}) // no value to send
3go func() { c <- collatz(n, done) }()
4select {
5case v := <-c:
6   fmt.Println(v)
7case <-time.After(20 * time.Millisecond):
8   fmt.Println("timed out")
9   close(done)
10}

Fun fact: many real-world closes broadcast when something is finished, like this one.


1func collatz(n int, done chan struct{}) int {
2   count := 0
3   for n > 1 {
4      select {
5      case <-done:
6         return -1
7      default:
8      }
9      if n%2 == 0 {
10         n /= 2
11      } else {
12         n = 3*n + 1
13      }
14      count++
15   }
16   return count
17}

Contexts

Contexts carry values down the call chain.

They also carry a "doneness" signal.


1func collatz(ctx context.Context, n int) int {
2   count := 0
3   for n > 1 {
4      select {
5      case <-ctx.Done(): // closed when done
6         return -1
7      default:
8      }
9      // ... same
10   }
11}
20

Contexts for timeouts

Use context.Context for timeouts.

Contexts inherit timeouts and cancellations from parents.

cancel must always be called to clean up resources.

1c := make(chan int) // unbuffered
2ctx, cancel := context.WithTimeout(context.Background(), 20*time.Millisecond)
3defer cancel()
4go func() { c <- collatz(ctx, 7) }()
5fmt.Println(<-c)                       // prints -1 on timeout
21

Contexts for real

What a "real" function might look like.

Where there is Context, there is almost always error.

1func collatz(ctx context.Context, n int) (int, error) {
2   count := 0
3   for n > 1 {
4      select {
5      case <-ctx.Done():
6         return 0, ctx.Err()
7      default:
8      }
9      if n%2 == 0 {
10         n /= 2
11      } else {
12         n = 3*n + 1
13      }
14      count++
15   }
16   return count, nil
17}

1func collatzWithTimeout(ctx context.Context, n int, tm time.Duration) (int, error) {
2   ctx, cancel := context.WithTimeout(ctx, tm)
3   defer cancel()
4   return collatz(ctx, n)
5}

Exercise: Hedging

Your server has two methods of performing a computation, or two backends that it can query for a result.

You could try one at random (a kind of load-balancing):

1func getResult(input string) string {
2   if rand.Int()%2 == 0 {
3      return method1(input)
4   } else {
5      return method2(input)
6   }
7}

Or you could try them both in parallel, and take the first result you get. This is hedging.

Start with getResult in exercises/hedging, and modify it to implement hedging.

  • Your getResult function should call both method1 and method2 concurrently.
  • It should return the first result it gets.
  • Before returning, it should cancel the other computation.
