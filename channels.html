Passing a value between goroutines

We can pass a value between goroutines with a WaitGroup.

1var wg sync.WaitGroup
2var v int
3wg.Go(func() { v = compute(7) })
4wg.Wait()
5fmt.Println(v)

But there is a more flexible way: channels.

2

Unbuffered channels

An unbuffered channel lets two goroutines rendezvous.

1c := make(chan int) // create a channel

2go func() { c <- compute(7) }() // send to c

3v := <-c // receive from c

4fmt.Println(v)

It doesn't matter which happens first, the send or the receive.

Sends and receives are thread-safe.

3

Multiples

You can have many senders, and many receivers.

1c := make(chan int)
2for i := range 5 {
3    go func() { c <- compute(i) }()
4}
5for range 5 {
6    go func() {
7        fmt.Println(<-c)
8    }()
9}

Goroutine leaks

1c := make(chan int)
2go func() { c <- compute(7) }()
3select {
4case v := <-c:
5    fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7    fmt.Println("timed out")
8}
// program continues
What happens to the first goroutine if there is a timeout?
  1. time.After case executes
  2. select finishes
  3. goroutine tries to send to c
  • The GC does not collect c: there is still a reference to it.
  • The GC does not collect goroutines: they must terminate.

Buffered channels

1c := make(chan int, 1)
2go func() { c <- compute(7) }()
3select {
4case v := <-c:
5    fmt.Println(v)
6case <-time.After(20 * time.Millisecond):
7    fmt.Println("timed out")
8}
// program continues
  • A channel can have a queue of values.
  • Sending enqueues, blocks if full.
  • Receiving dequeues, blocks if empty.
  • Sender and receiver don't have to rendezvous.
Now what happens to the first goroutine if there is a timeout?
  1. time.After case executes
  2. select finishes
  3. goroutine tries to send to c
  4. value is enqueued in channel
  5. goroutine exits

No leaks: goroutine terminates

No garbage: Nothing refers to c

Non-blocking select

Let's implement an in-process notification service:

// sendNotification optionally sends a notification without
// blocking the current goroutine.
// To avoid blocking, the notification might be dropped.
1func sendNotification(string) {}

// receiveNotification receives a notification if there is one.
// It returns "" if there isn't.
// It never blocks.
2func receiveNotification() string { return "" }
13

Notifications: first attempt

1var nc = make(chan string)

2func sendNotification(s string) { nc <- s }

3func receiveNotification() string { return <-nc }
Does this implementation satisfy our spec?

No. Both functions can block.

What if we add buffering?

It helps until the buffer fills (send) or is empty (receive). Then they block again.

14

Notifications: solution

The default case of a select executes when no channel operations are ready.

1var nc = make(chan string, 10)

2func sendNotification(s string) {
3    select {
4    case nc <- s:
5    default: // if we can't send, drop s
6    }
7}

8func receiveNotification() string {
9    select {
10    case s := <-nc:
11        return s
12    default:
13        return ""
14    }
15}

Closing channels

Close a channel when it will never be sent to again.

1type node struct {
2    val         int
3    left, right *node
4}

5func sendValues(n *node, ch chan int) {
6    if n == nil {
7        return
8    }
9    sendValues(n.left, ch)
10    ch <- n.val
11    sendValues(n.right, ch)
12}

(In modern Go, we would use an iterator.)

1func printTree(root *node) {
2    c := make(chan int)
3    go func() {
4        sendValues(root, c)
5        close(c)
6    }()

7    for {
8        v, ok := <-c // distinguish closed from zero value
9        if !ok {
10            // c is closed
11            break
12        }
13        fmt.Println(v)
14    }
15}

close broadcasts

close affects every receiver

1c := make(chan int)
2var wg sync.WaitGroup
3wg.Go(func() { c <- 1 })
4wg.Go(func() { fmt.Println(<-c) })
5wg.Go(func() { fmt.Println(<-c) })
6wg.Wait()
What does this do?

Print 1, then hang.

1c := make(chan int)
2var wg sync.WaitGroup
3wg.Go(func() { close(c) })
4wg.Go(func() { fmt.Println(<-c) })
5wg.Go(func() { fmt.Println(<-c) })
6wg.Wait()
What does this do?

Print 0 twice, then finish.

compute(7) will keep running until it's done, consuming resources.

You can't interrupt an arbitrary goroutine.

It has to check.

Asking a goroutine to stop

1c := make(chan int, 1)
2done := make(chan struct{}) // no value to send
3go func() { c <- compute(7, done) }()
4select {
5case v := <-c:
6    fmt.Println(v)
7case <-time.After(20 * time.Millisecond):
8    fmt.Println("timed out")
9    close(done)
10}

Fun fact: most real-world closes broadcast when something is finished, like this one.

1func compute(x int, done chan struct{}) int {
2    t := 0
3    for {
4        select {
5        case <-done:
6            return -1
7        default:
8            x, ok := computeALittle(t)
9            if !ok {
10                return t
11            }
12            t += x
13        }
14    }
15}

Contexts

Use context.Context for timeouts.

1c := make(chan int) // unbuffered
2ctx, cancel := context.WithTimeout(
3    context.Background(),
4    20*time.Millisecond)
5defer cancel() // always defer cancel
6go func() { c <- compute(ctx, 7) }()
7fmt.Println(<-c) // prints -1 on timeout
1func compute(ctx context.Context, x int) int {
2    t := 0
3    for {
4        select {
5        case <-ctx.Done():
6            return -1
7        default:
8            x, ok := computeALittle(t)
9            if !ok {
10                return t
11            }
12            t += x
13        }
14    }
15}
21

Contexts, continued

What a "real" function might look like.

1func computeWithTimeout(ctx context.Context,
2    tm time.Duration, arg int,
3) (int, error) {
4    c := make(chan int, 1)
5    ctx, cancel := context.WithTimeout(ctx, tm)
6    defer cancel()
7    go func() { c <- compute(ctx, arg) }()
8    select {
9    case v := <-c:
10        return v, nil
11    case <-ctx.Done():
12        return 0, ctx.Err()
13    }
14}
Do we still need a buffered channel?

Yes. If the second select case is taken, the goroutine would be blocked forever sending to c, even if compute returns early.

Does the select need a default case?

No. The only two possibilities are that compute finishes on time and sends to c, or that the context times out and closes its Done channel.

Exercise: Hedging

Your server has two methods of performing a computation, or two backends that it can query for a result.

You could try one at random (a kind of load-balancing):

1func getResult(input string) string {
2    if rand.Int()%2 == 0 {
3        return method1(input)
4    } else {
5        return method2(input)
6    }
7}

Or you could try them both in parallel, and take the first result you get. This is hedging.

Start with the code above, in TODO, and modify it to implement hedging.

  • Your getResult function should call both method1 and method2 concurrently.
  • It should return the first result it gets.
  • Before returning, it should cancel the other computation.
